# RFC: MiMo-V2.5 OMNI 全模态实现方案

**状态**: Draft
**作者**: john
**日期**: 2026-05-20
**模型**: XiaomiMiMo/MiMo-V2.5 (text/image/video/audio → text)

---

## 1. 背景

MiMo-V2.5 是小米最新的 OMNI 全模态模型，支持文本、图片、视频、音频输入，文本输出。
由三个核心组件构成：

1. **LLM Backbone** — 48 层 hybrid SWA/full-attention MoE 模型，使用 fused FP8 量化 QKV 权重
2. **Vision Encoder** — 自定义 ViT，Conv3D patch embedding、GQA、hybrid window attention、attention sinks
3. **Audio Encoder** — RVQ codebook → speech embeddings → 6 层 Qwen2 transformer → MLP projection

模型采用 **early fusion** 方案：输入序列中的占位 token（`<|image_pad|>`、`<|video_pad|>`、audio token）
在进入 LLM backbone 前被替换为对应 encoder 的输出 embedding。

### 1.1 架构概览

| 组件 | 架构 | 关键参数 |
|------|------|---------|
| LLM Backbone | 48 层 hybrid MoE（9 full-attn + 39 SWA） | 64 heads，full: 4 KV heads，SWA: 8 KV heads |
| Vision Encoder | 28-block ViT，Conv3D patch embed | 32 heads / 8 KV heads (GQA)，hidden=1280，out=4096 |
| Audio Encoder | RVQ → 20-ch speech embeddings → 6 层 Qwen2 → MLP | input_local_dim=1024，out=4096 |
| Fusion | Early fusion (`_replace_modal_embeddings_inplace`) | image_token_id=151655，video_token_id=151656，audio_token_id=151669 |

### 1.2 与现有模型的关键差异

| 特性 | MiMo-V2.5 | Qwen2.5-VL | Qwen3-Omni (Ming) |
|------|-----------|------------|-------------------|
| ViT attention | 三类 hybrid：full(-1) / row-window(0) / col-window(1) | Full + window (2D) | Full + window (2D) |
| ViT GQA | 是（32h / 8kv） | 否（MHA） | 否（MHA） |
| Attention sinks | 是（use_sink=True） | 否 | 否 |
| Patch embed | Conv3D（temporal_patch_size=2） | Conv3D | Conv3D |
| Audio encoder | RVQ → Qwen2 transformer → MLP | 无 | Whisper → Conv1d |
| LLM 量化 | Fused QKV FP8（per-TP-shard block-wise） | FP16/BF16 | FP16/BF16 |
| LLM attention | Hybrid SWA/full-attn（不同层 KV heads 数不同） | Full | Full |

## 3. 待实现工作项

### P1: MiMo-V2.5 Vision Encoder

**复杂度**: 高
**预估工作量**: 3-5 天
**路径**: `python/sgl_jax/srt/multimodal/models/mimo_v2_5/mimo_vision_encoder.py`

Vision encoder 是最大的新组件。虽然结构上与 Qwen2.5-VL / Qwen3-Omni ViT 相似，
但 MiMo-V2.5 有三个独特特性需要新实现：

#### 3.1.1 Hybrid Window Attention

MiMo-V2.5 ViT 使用三种 attention 类型，由 `vit_window_attn_types` 逐 block 控制：

| 类型 | 值 | 行为 |
|------|-----|------|
| Full attention | -1 | 标准全局 attention（block 0、9、18、27） |
| Row-window | 0 | 行方向窗口内 attention，`window_size=128` tokens |
| Column-window | 1 | 列方向窗口内 attention，`window_size=128` tokens |

需要自定义 attention mask builder，按空间位置划分 row/column 窗口。
HF 参考实现见 `MiMoVisionAttention`（modeling_mimo_v2.py:645）。

**方案**：实现 `MiMoVisionAttention`，接受 `window_type` 参数。对于 row/column window，
根据空间网格维度计算窗口划分，构建 block-diagonal attention mask；full attention block 使用标准 attention。

#### 3.1.2 GQA（Grouped Query Attention）

与 Qwen2.5-VL（MHA，16/16 heads）和 Qwen3-Omni（MHA，16/16 heads）不同，MiMo-V2.5 ViT 使用 GQA，
32 个 query head、8 个 KV head（`num_query_groups=4`）。

**方案**：在 `VisionAttention` 中加入 KV head repeat（与 LLM GQA 相同逻辑）。Qwen3-Omni 的
`vision_encoder.py` 已支持可配置 `num_key_value_heads`，可作为基础。

#### 3.1.3 Attention Sinks

当 `use_sink=True` 时，第一个 token 的 attention score 在所有窗口中被保留（充当全局信息聚合点）。
仅在 windowed attention block 中生效。

**方案**：在 windowed attention 中，每个窗口的 attention scope 始终包含 position 0。

#### 3.1.4 其他组件（可复用）

| 组件 | 参考 | 复用策略 |
|------|------|---------|
| Conv3D PatchEmbed | Qwen3-Omni `Vision3DPatchEmbed` | 直接复用（结构相同） |
| 2D RoPE | Qwen2.5-VL `VisionRotaryEmbedding` | 直接复用 |
| SwiGLU MLP | Qwen3-Omni `VisionMLP` | 直接复用（SiLU 激活，同模式） |
| PatchMerger | Qwen2.5-VL `VisionPatchMerger` | 直接复用（`spatial_merge_size=2`） |
| 权重加载 | Qwen3-Omni 模式 | 适配 V2.5 权重名称 |

### P2: MiMo-V2.5 Audio Encoder

**复杂度**: 中
**预估工作量**: 2-3 天
**路径**: `python/sgl_jax/srt/multimodal/models/mimo_v2_5/mimo_audio_encoder.py`

MiMo-V2.5 的 audio encoder **不是**基于 Whisper 的（与 Ming-Omni 不同），流程如下：

1. **Speech embeddings**：RVQ codebook（1280 vocab）→ 20 channel embeddings → 分组（group_size=4）
   → 5 组 embeddings 拼接
2. **Transformer**：6 层 Qwen2 风格 transformer，RoPE，hidden_dim=1024，16 local attention heads
3. **Projection**：2 层 MLP，从 1024 投影到 4096（LLM hidden_size）

**现有资产**：
- `mimo_audio_tokenizer.py` — RVQ codec（raw audio → discrete codes），可复用于音频 tokenization
- `mimo_audio_backbone.py` — Qwen2 风格 transformer（6 层），架构与 V2.5 audio encoder 的 transformer block 非常相似

**方案**：参考 modeling_mimo_v2.py:932 的 `MiMoAudioEncoder` 构建 `MiMoV25AudioEncoder`。
speech embedding → transformer → projection 流水线结构清晰。主要新工作是实现 speech embedding 层
（RVQ codebook lookup + channel 分组）。

**与 MiMo-Audio 的关键区别**：V2.5 audio encoder 是 OMNI 模型的内部组件，不是独立模型。
它接收预 tokenize 的 audio codes（来自外部 tokenizer，可能是同一个 `MiMoAudioTokenizer`），
输出 embeddings 供 LLM fusion 使用。

### P3: OMNI Forward Path（Embedding Fusion）

**复杂度**: 中
**预估工作量**: 1-2 天
**路径**: `python/sgl_jax/srt/models/mimo_v2_5_omni.py`（或扩展 `mimo_v2_pro.py`）

OMNI forward path 将占位 token 替换为 encoder 输出：

```
input_ids: [text... <|image_pad|>×N ... text... <|audio_pad|>×M ... text...]
                     ↓                          ↓
              vision_embeds (N×4096)     audio_embeds (M×4096)
                     ↓                          ↓
input_embeds: [text_embed... vision_embeds... text_embed... audio_embeds... text_embed...]
```

**实现步骤**：
1. 在 embedding stage（Stage 0）中，对文本 token 调用 `get_input_embeddings()`
2. 对每个有占位 token 的模态：
   - 图片：替换 `image_token_id=151655` 位置为 vision encoder 输出
   - 视频：替换 `video_token_id=151656` 位置为 vision encoder 输出
   - 音频：替换 `audio_token_id=151669` 位置为 audio encoder 输出
3. 将融合后的 embeddings 传入 LLM backbone（Stage 1），设置 `input_embedding=True`

**参考**：modeling_mimo_v2.py:554 的 `_replace_modal_embeddings_inplace`，
modeling_mimo_v2.py:1758 的 `_get_multimodal_embeds`。

sgl-jax 中 Qwen3-Omni 的 `Qwen3OmniMoeThinkerEmbedding` 已有相同模式。

### P4: Stage Pipeline 配置

**复杂度**: 低
**预估工作量**: 0.5-1 天
**路径**: `python/sgl_jax/srt/multimodal/models/static_configs/mimo_v2_5_omni_stage_config.yaml`

定义 MiMo-V2.5 OMNI 的 SJMRT stage pipeline：

```yaml
model_arch: MiMo-V2.5-OMNI
stage_args:
  # Stage 0: Embedding — vision/audio 编码 + token 融合
  - stage_id: 0
    model_class: MiMoV25OmniEmbedding
    scheduler: embedding
    runtime:
      num_tpus: 1
      device_kind: cpu  # 或 tpu（ViT 较大时）
      max_batch_size: 1

  # Stage 1: LLM Backbone — prefill + decode
  - stage_id: 1
    model_class: MiMoV2ForCausalLM  # 已有
    scheduler: auto_regressive
    runtime:
      num_tpus: 16  # v6e-16 上 TP=16
      max_batch_size: 1
    precompile_params: {"input_embedding": true}
    final_output: true
```

**需要的注册**：
- 在 `get_model_class()` 中注册 `MiMoV25OmniEmbedding`
- 如果 vision 作为单独 stage 运行，需注册 `MiMoV25VisionEncoder`

**设计决策**：vision 和 audio 是否在 Stage 0 中一起编码（CPU/单 TPU），还是拆成独立 stage。
建议：先用单一 Stage 0（最简），后续按延迟需求再拆分。

### P5: 权重加载与模型注册

**复杂度**: 中
**预估工作量**: 1-2 天

具体任务：
1. **Vision encoder 权重**：映射 HF 权重名称（`visual.*`）到 sgl-jax ViT 参数。
   在 LLM backbone 权重加载中跳过 vision 权重（在 `mimo_v2_pro.py` 中添加 `visual.` 到 skip prefixes）。
2. **Audio encoder 权重**：类似地映射 HF 权重名称（`audio_encoder.*`）。
3. **模型配置解析**：扩展 `ModelConfig`，从 V2.5 的 `config.json` 中解析 `vision_config` 和 `audio_config`。
4. **Processor 配置**：处理 `preprocessor_config.json`（Qwen2_5_VLProcessor 兼容），
   用于图像预处理（min/max pixels、patch_size、merge_size、归一化）。

### P6: 端到端验证

**复杂度**: 中
**预估工作量**: 2-3 天

1. **纯图片测试**：单张图片 + 文本 prompt，验证 LLM 生成正确描述
2. **视频测试**：短视频片段，验证时序理解能力
3. **音频测试**：音频片段 + 文本 prompt，验证音频理解能力
4. **混合模态测试**：图片 + 音频 + 文本在单个 prompt 中
5. **Logit 对比**：与 HF 参考实现对比相同输入的 first-token logits

## 5. 范围决策

### 5.1 音频：建议实现（优先级低于视觉）

**建议**：实现 audio encoder，但优先做 vision。

理由：
- V2.5 权重中包含 `audio_encoder.*` 参数——模型本身具备该能力
- Audio encoder 架构简单（RVQ embeddings → 小 transformer → MLP）
- 现有 `mimo_audio_tokenizer.py` 已处理 codec 层
- 如果时间紧张，audio 可推迟到第二阶段

### 5.2 不在范围内

| 项目 | 原因 |
|------|------|
| 音频输出（TTS/vocoder） | 当前 scope 为纯文本输出 |
| SJMRT 大规模重构 | 侧重 modeling，不做大的调用结构 refactor |
| 动态分辨率（任意图片尺寸） | 先用固定分辨率，后续再加 |
| 流式/分块音频输入 | 先用完整音频编码，后续优化 |
